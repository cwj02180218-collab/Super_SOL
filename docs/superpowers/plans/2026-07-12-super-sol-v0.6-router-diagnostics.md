# Super SOL v0.6 Router Diagnostics Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a free-testable `0.6.0rc1` candidate that separates routing quality from procedure quality and repairs the observed failure-atomicity recall gap without overfitting disclosed task IDs.

**Architecture:** Keep the stock-Codex plugin local and single-turn. Add environment-only diagnostic controls so benchmark homes can observe routing or force a preregistered route without changing the model-visible prompt; expand general failure-atomicity signals; freeze a four-arm diagnostic protocol before any paid run.

**Tech Stack:** Python 3.12 package, standard-library Python 3.9 hook runtime, pytest, Ruff, basedpyright, uv, stock Codex plugin hooks.

## Global Constraints

- Package version is `0.6.0rc1`; plugin version is `0.6.0-rc1`.
- No API, model, subagent, Docker grader, or billable benchmark call is allowed in implementation.
- Never add T114, checkout, config-precedence, or benchmark-answer strings to the router or packs.
- Normal users get adaptive mode. Diagnostic controls are environment-only and never alter the prompt.
- `SUPER_SOL_DIAGNOSTIC_MODE` accepts only `observe` or `forced`; other values fail closed to adaptive behavior with a private warning state.
- `SUPER_SOL_FORCED_ROUTE` is honored only in forced mode and only for a non-pass-through public route.
- Observe mode records the natural route but emits no `additionalContext` and no repair context.
- Forced mode emits one frozen pack and one evidence-triggered repair at most; it does not bypass billing guards.
- Public claims remain blocked until a new externally sealed holdout passes.
