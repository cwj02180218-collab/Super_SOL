# Super SOL v0.4 Lean design

## Decision

Super SOL v0.4 will replace broad procedural guidance with one bounded contract sweep. The plugin
must first prove that it is no worse than raw Codex on quality while staying within explicit token
and latency budgets. A release candidate that misses any promotion gate falls back to the existing
observation-only behavior and cannot advertise a performance benefit.

The design optimizes for the following order:

1. Do not reduce task quality.
2. Detect requirement semantics that visible tests leave uncovered.
3. Keep additional tokens and time tightly bounded.
4. Preserve the existing privacy, billing, and beginner-facing safety boundaries.

## Evidence behind the redesign

The provisional four-arm benchmark produced 26 valid results out of 32 scheduled slots. On eight
paired Codex tasks, the measured Super SOL arm was 8.5 score points below raw Codex, used 98,923
more tokens on average, and took 23.7 seconds longer. Three paired tasks accounted for every score
regression:

- T105 used a shallow defensive copy and missed nested aliasing.
- T106 returned the right error code for an unknown command but missed the required usage message.
- T107 suppressed a duplicate item after its first attempt failed instead of retrying it once.

All three implementations passed their visible tests. This shows a real product weakness: v0.3.1
checks for observed verification but does not help Codex cover every semantic clause in the request.

The same experiment cannot establish the size of the Super SOL effect because the two Codex arms
were not isolated symmetrically. The raw arm used `--ignore-user-config`; the Super SOL arm loaded
the user's complete Codex configuration and all enabled plugins. Some enhanced runs also read Super
SOL from more than one cached path. Large token deltas were concentrated in a few tasks rather than
appearing as a stable hook cost. The existing result remains a valid regression signal, but it is
not a clean causal estimate of the plugin alone.

## Goals

- Preserve automatic beginner-friendly operation after installation.
- Add at most one compact model-visible instruction for actionable requests.
- Reduce hidden semantic omissions without exposing hidden tests or benchmark-specific answers.
- Prevent Super SOL from triggering plans, skill chains, repeated tests, model continuations,
  subagents, model changes, or billable API calls.
- Provide a clean-room Codex A/B runner whose only arm difference is the Super SOL plugin.
- Promote the lean behavior only after a preregistered unseen holdout meets non-inferiority and
  efficiency gates.

## Non-goals

- Changing the intrinsic capability of a model.
- Copying Fablize's Claude-specific implementation or claiming Fable parity.
- Optimizing directly against the known private assertions from T105 through T107.
- Automatically selecting a model or reasoning effort.
- Automatically retrying a task, starting a second model turn, or creating a subagent.
- Treating visible test success as proof that every requested behavior is correct.
- Publishing a performance claim from the tuning set.

## Considered approaches

### Observation only

Keep only mutation and successful-verification observation. This has the lowest likely overhead and
is the mandatory fallback, but it cannot address the semantic omissions found in the benchmark.

### Lean contract sweep

Inject one short instruction that asks Codex to silently map request clauses to code paths and
inspect one applicable boundary after visible tests pass. This is the selected candidate because it
targets the observed weakness without adding a second pass or a large workflow.

### Adaptive second pass

Trigger a separate review or continuation for ambiguous tasks. This may increase the capability
ceiling, but it conflicts with the no-automatic-call and efficiency requirements. It is deferred
until a lean single-pass design has credible holdout evidence.

## Runtime architecture

### Hook surface

The automatic plugin retains only the lifecycle hooks needed for safety and evidence:

- `UserPromptSubmit`: local classification and at most one compact context injection.
- `PreToolUse`: existing explicit-billing protections.
- `PostToolUse`: existing local mutation and verification evidence.
- `Stop`: existing one-time warning when a mutation lacks a later successful verification.

The `SessionStart` context hook is removed. Its generic instructions duplicate normal Codex
behavior on every task and are not needed for billing enforcement or evidence tracking.

### Skill surface

The Super SOL skill remains available for explicit questions about the harness, but
`allow_implicit_invocation` becomes `false`. Normal implementation and debugging tasks receive the
lean hook context without making Codex discover and read the full Super SOL skill. This prevents the
plugin from becoming an entry point to unrelated procedural skills.

The plugin does not attempt to disable platform or developer-mandated skills. The clean-room runner
instead holds those surfaces constant across both arms.

### Contract sweep

Action and debug requests receive this exact 154-character English instruction once:

> Silently map requirements to code and one boundary. After tests pass, re-read once for ownership,
> input, state, and failure semantics. Do not rerun tests.

This text is deliberately model-facing rather than user-facing, language-neutral for benchmark
prompts, and short enough to make its context cost auditable. The implementation stores it as one
constant and tests both its exact value and word count.

The four semantic families are general contracts, not answers to specific benchmark tasks:

- Ownership: aliasing, shallow or deep copying, and caller mutation.
- Input partition: unknown input, arity, exhaustive branches, and return or output contracts.
- State transition: retry, duplicate handling, order, idempotency, and checkpoint timing.
- Failure semantics: partial failure, atomicity, cancellation, and interruption propagation.

The model is told to inspect only applicable families. It must not enumerate the checklist in its
answer, create extra artifacts, or repeat a passing verification solely because of Super SOL.

### Classification behavior

- `conversation`: no additional model context and no verification expectation.
- `action`: contract sweep once.
- `debug`: the same contract sweep once; reproduce only when an existing failure is supplied.
- `release`: retain one compact release-readiness context, without the contract sweep.

Classification remains deterministic and local. Prompt text is not persisted.

### Evidence behavior

Super SOL continues to recognize only structured local tool outcomes. A successful verification
must occur after the latest mutation. The contract sweep is advice, not a second success gate,
because the plugin cannot truthfully observe a model's private reasoning.

No model-visible context, prompt, command, output, path, or environment value is added to plugin
state. State continues to contain only hashed turn identity, classification, timestamps, billing
authorization, and mutation or verification booleans.

## Clean-room Codex A/B runner

### Isolation contract

The runner creates one minimal base `CODEX_HOME`, then derives two homes from the same snapshot:

- `raw`: no Super SOL marketplace or plugin.
- `lean`: exactly one digest-recorded Super SOL marketplace and one enabled Super SOL plugin.

It never copies the user's normal Codex configuration, plugin cache, memories, session logs, MCP
configuration, or shell environment into either arm. Authentication remains outside benchmark
artifacts and is never serialized by the runner.

Both arms use the same:

- absolute Codex executable and version;
- model and reasoning effort;
- task prompt, AGENTS instructions, workspace content, and file modes;
- permissions and hook-trust command flags;
- timeout, environment allowlist, and telemetry parser;
- platform-provided system skills and bundled runtime.

Before any paid or quota-consuming execution, preflight records both plugin lists, recursively
hashes both home trees, and compares them against an allowlist. The only permitted tree difference
is the Super SOL marketplace, plugin, and its generated local state directory. The runner fails
closed if it sees another enabled plugin, more than one Super SOL `SKILL.md`, a mutable marketplace
reference, or any unexpected configuration difference.

The asymmetric `--ignore-user-config` invocation used by the provisional benchmark is forbidden.
Both arms must use the same configuration-loading mode.

### Provenance

Every session records:

- Codex binary path, version, and binary digest;
- model and reasoning effort;
- task, fixture, visible-test, AGENTS, and prompt digests;
- raw and lean `CODEX_HOME` tree digests;
- exact plugin tag and plugin tree digest;
- command argument digest with secret values excluded;
- wall time and input, cache-read, and output tokens;
- changed-file list, public grade, hidden grade, and infrastructure status.

Authentication data, raw prompts containing secrets, model workspaces, and hidden tests are not
published. Aggregate evidence is published only after leakage review.

### Failure handling

- Rate limits, session limits, timeouts, and provider errors are infrastructure-missing outcomes,
  never score zeroes.
- A missing artifact, provenance mismatch, duplicate session, or parser error invalidates the run.
- Interrupted slots may resume only with the same preregistered identity.
- The runner never retries a model slot automatically unless the user has separately approved the
  resumed billable or quota-consuming batch.

## Evaluation program

### Gate 0: free static and local validation

Gate 0 makes no model call. It requires:

- all existing project tests;
- exact hook payload golden tests;
- no context for conversation requests;
- one context injection for action and debug requests;
- exact contract-sweep copy and bounded word count;
- no `SessionStart` model context;
- `allow_implicit_invocation: false`;
- no automatic continuation, subagent, model change, or API call;
- temporary-home install, list, remove, and duplicate-path checks;
- clean-room dry-run provenance and configuration-diff checks.

### Gate 1: tuning regression set

T105 through T108 are run twice per arm in crossover order: four tasks by two arms by two
repetitions, for 16 slots. T105 through T107 test the known semantic failures; T108 detects the
previous token spike. These tasks are a tuning and regression set only. Their result cannot support
a public performance claim.

Gate 1 requires a fresh explicit approval before any quota-consuming Codex execution. A failed gate
returns the candidate to observation-only mode without proceeding to holdout.

### Gate 2: unseen confirmatory holdout

T109 through T116 are eight new tasks whose hidden grader pack is sealed before Gate 1 completes.
Each task runs twice per arm in balanced crossover order, for 32 slots. Neither hidden tests nor
grader feedback may influence the contract-sweep text or implementation after preregistration.

Only Gate 2 may support a v0.4 performance statement.

## Promotion gates

The lean behavior becomes the v0.4 default only if all conditions hold on Gate 2:

- paired mean score uplift is at least 0 points;
- the paired bootstrap 95% confidence lower bound is at least -2 points;
- full hidden-test pass rate is at least the raw arm's rate;
- mean total-token increase is no more than 5%;
- mean wall-time increase is no more than 10%;
- no task repeats a score regression of 10 points or more across both repetitions;
- artifact omissions and hidden-test leakage are both zero;
- an independent audit reproduces the aggregate report from raw records.

Sign-test and paired rank-biserial results are reported but are not used alone to claim proof with a
small sample. Provider-reported cost is reported when available and never imputed for Codex.

## Fallback and release policy

- v0.3.1 remains the immutable stable release during development.
- v0.4.0-rc1 contains the lean candidate and free Gate 0 evidence.
- Gate 1 must pass before Gate 2 may start.
- Gate 2 must pass before the lean contract sweep becomes the default in v0.4.0.
- If any promotion condition fails, v0.4 ships observation-only behavior or does not ship.
- README performance language is prohibited until the confirmatory report and audit are public.

Rollback never rewrites an existing tag. A failed candidate receives a new corrective tag or is
withdrawn before stable promotion.

## Test strategy

### Hook unit tests

- action and debug profiles emit the exact 154-character contract sweep once;
- conversation emits no context;
- release emits only its compact release context;
- prompt classification and secret blocking remain deterministic;
- mutation and verification evidence ordering remains unchanged;
- Stop never starts another model turn;
- billing guards retain their exact confirmation and CLI-flag requirements.

### Plugin contract tests

- the manifest contains no network client, MCP server, or app;
- implicit skill invocation is disabled;
- `SessionStart` is absent from the registered hook events;
- one clean installation contains exactly one Super SOL skill path;
- install, list, update, remove, and reinstall work in an isolated `CODEX_HOME`.

### Clean-room runner tests

- fixture homes differing only by Super SOL pass preflight;
- an extra plugin, config key, skill path, or mutable ref fails preflight;
- dry-run creates no model process;
- infrastructure failures remain missing rather than becoming scores;
- session identity changes when any task, environment, plugin, model, or effort input changes;
- external grades bind to the exact session identity;
- resume accepts only preregistered missing slots.

## Documentation changes

The release documentation must state:

- the provisional v0.3.1 benchmark observed a negative signal but had asymmetric Codex config;
- v0.4 is a non-inferiority candidate, not a proven capability boost;
- the existing plugin does not make automatic billable calls;
- the optional clean-room benchmark requires explicit approval and may consume Codex quota;
- tuning-set results and confirmatory holdout results are reported separately.

## Security review

The redesign does not broaden filesystem, process, network, billing, or model authority. The new
runner launches Codex only after explicit approval, uses an environment allowlist, writes under an
ignored benchmark root, excludes credentials from artifacts, and treats hooks as quality controls
rather than an operating-system security boundary.
