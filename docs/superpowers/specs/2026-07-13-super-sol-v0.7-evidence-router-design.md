# Super SOL v0.7 Evidence-Bounded Router Design

**Status:** Approved for implementation  
**Target release:** `v0.7.0-rc1`, promoted to `v0.7.0` only after the frozen holdout gate  
**Base:** Super SOL `v0.6.0-rc1` source head `3a5f6fdb06ae555753867f6819f4d91716aeb32d`

## 1. Problem statement

The six-cell Orca crossover completed 96 valid slots on T109-T116 with two repetitions per
cell. Super SOL v0.6 was inexpensive compared with LazyCodex, but it did not establish a
reliable uplift:

| Comparison | Mean score delta | 95% bootstrap CI | Full pass | Token ratio | Time ratio |
| --- | ---: | ---: | ---: | ---: | ---: |
| Terra + Super SOL vs raw | -0.71 | [-6.09, 4.51] | 8/16 vs 9/16 | 1.07x | 1.12x |
| Sol + Super SOL vs raw | +2.53 | [-1.28, 6.70] | 12/16 vs 10/16 | 1.07x | 1.29x |

The task-level pairs explain why a new design is required:

- T109 improved by +17.5 twice on both models, but the task was naturally pass-through.
- The concurrency route did not improve T112 and added overhead.
- The migration route sometimes helped T113 and sometimes regressed by 13.06 points.
- The security route regressed Terra T115 by 20 points twice.
- Every v0.6 comparison exceeded the desired 5% token budget; Sol also exceeded the 10%
  time budget.

The failure is not a lack of procedure volume. Up-front specialist packs can displace a good
raw solution, widen the patch, and cause the model to optimize for a procedure instead of the
literal request. v0.7 therefore preserves raw reasoning first and spends its intervention budget
only after observed verification.

## 2. Goals and non-goals

### Goals

1. Preserve the raw Codex implementation path until the first observed verification result.
2. Catch one explicit but still-uncovered semantic obligation after a successful verification.
3. Never create an unbounded planning, review, retry, subagent, or test loop.
4. Keep model-visible context bounded, deterministic, privacy-minimized, and auditable.
5. Promote to stable only when both Terra and Sol satisfy quality and cost gates on a frozen
   post-freeze holdout.
6. Publish score, pass-rate, token, time, uncertainty, and task-pair visualizations with the
   release evidence.

### Non-goals

- Claiming that a plugin changes a model's intrinsic intelligence.
- Switching models or reasoning effort.
- Automatically invoking a subagent, external API, MCP server, or second model turn.
- Inspecting or storing prompt text, source content, tool output, model output, environment
  values, or credentials.
- Re-running a successful test merely because Super SOL is installed.
- Promoting a point estimate whose confidence interval or regression guard fails.

## 3. Approaches considered

### A. Larger specialist packs

Add more routes and more detailed checklists before implementation. This can raise the ceiling on
some tasks, but v0.6 already demonstrated route-specific regressions and 7% token overhead. More
up-front context increases interference and is rejected.

### B. Multi-agent adaptive review

Have a reviewer model inspect the first solution. This offers a high ceiling but changes the model
topology, creates billable calls, and resembles the LazyCodex behavior that consumed 19x-38x
tokens without improving average quality. It is rejected for the default plugin.

### C. Raw-first residual contract check

Let stock Codex implement and verify normally. After the first successful verification following a
change, inject one short reminder naming the highest-confidence requested contract. The model
compares its patch against the request once, makes at most one focused correction if needed, and
does not repeat a passing test. This is the selected approach.

## 4. Runtime architecture

### 4.1 Contract router

`super_sol_routes.py` replaces broad route packs with deterministic contract metadata:

- `ownership_aliasing`
- `input_error_semantics`
- `retry_state`
- `concurrency_cancellation`
- `failure_atomicity`
- `migration_compatibility`
- `security_path_boundary`

Each signal has an identifier, weight, phrases, and a contract family. The router returns:

- the highest-confidence primary contract or no contract;
- at most one secondary signal identifier for audit only;
- a confidence score;
- whether a diagnostic control forced the decision.

Ambiguous prompts no longer activate several packs or fall through silently. The highest score
wins only when it reaches the activation threshold and beats the runner-up by the configured
margin. Ties are observe-only. Explicit `SUPER SOL OFF` and diagnostic controls remain supported.

### 4.2 Raw-first request handling

`UserPromptSubmit` stores only hashed turn identifiers and privacy-safe routing metadata. Adaptive
mode emits no model-visible context. Forced diagnostic mode may still emit a named legacy pack so
router and procedure effects remain separately measurable, but this is never the default.

### 4.3 Evidence ledger

The hook records immutable bounded events for:

- a recognized edit tool;
- a recognized verification command and exit code;
- whether the residual check was claimed;
- whether a failed-verification repair cue was claimed.

Raw command text, tool output, file names, source text, and prompt text are never stored. State
files remain owner-only, atomically written, and bounded to 4096 bytes each.

### 4.4 One residual contract check

After all of the following are observed in one turn:

1. an action request with one high-confidence contract;
2. at least one recognized edit;
3. the first successful recognized verification;
4. no prior residual check claim;

`PostToolUse` emits exactly one context:

> Final semantic check only: compare the patch once with the requested `<contract>` behavior. If
> one explicit edge is missing, make one focused correction; otherwise stop. Do not rerun passed
> tests.

The rendered context must be at most 220 Unicode code points and contain no prompt excerpt. No
context is emitted for pass-through, explanation, ambiguous, observe-mode, no-edit, failed-test,
or repeated-success events.

### 4.5 Failed verification

A failed recognized verification after an edit may emit one short repair cue. It references only
the observed failure and active contract, asks for the smallest correction, and permits the same
focused verification once. A second failure emits no context. Successful tests are never repeated
by instruction.

### 4.6 Hard budgets

The default adaptive path enforces:

- maximum model-visible injections per turn: 1 total, either a residual check after success or a
  repair cue after failure;
- maximum residual context: 220 Unicode code points;
- automatic model calls: 0;
- automatic subagents: 0;
- model/effort changes: 0;
- hook wall time gate: absolute p95 below 125 ms and less than 90 ms above a same-runtime empty
  Python process; observed product target remains p95 below 100 ms;
- stored prompt/source/output bytes: 0;
- repeated successful verification instructions: 0.

## 5. Evaluation architecture

### 5.1 Development gate

Before candidate freeze:

- all existing 255 tests remain green;
- new route, state-machine, context-budget, privacy, and mutation tests pass;
- T109-T116 are used only as disclosed regression diagnostics;
- the prior 96-slot manifest is replayed through the router without model calls to audit route
  activation and expected context counts.

The candidate plugin commit is then frozen. Product code cannot change after the holdout task pack
is created. Any product change requires a new candidate and a new holdout.

### 5.2 Frozen holdout

After candidate freeze, create T117-T124 in a separate benchmark target commit. The tasks cover
the seven contract families in different application domains and include one neutral pass-through
task. Public tests and model-hidden semantic tests remain physically separated. The plugin and
model receive only TASK.md and public repository content.

Run a four-cell Orca crossover:

- `terra-raw`: GPT-5.6 Terra, medium
- `terra-v07`: GPT-5.6 Terra, medium + candidate
- `sol-raw`: GPT-5.6 Sol, high
- `sol-v07`: GPT-5.6 Sol, high + candidate

Eight tasks, two repetitions, 64 valid slots, balanced across 16 waves. Provider failures,
timeouts, and telemetry parser failures are censored and retried in new worktrees. Only the latest
valid observation enters paired statistics.

### 5.3 Stable promotion gate

`v0.7.0` requires every condition below for each model separately unless stated otherwise:

- mean score delta >= 0;
- bootstrap 95% CI lower bound >= -2 points;
- full-pass rate >= raw;
- aggregate token ratio <= 1.05;
- aggregate wall-time ratio <= 1.10;
- no repeated task regression <= -10 points;
- no leaked hidden test, missing artifact, provenance mismatch, or secret;
- 64/64 valid lattice after censored retries;
- plugin hook absolute p95 below 125 ms and incremental p95 below 90 ms over the measured empty
  Python process floor.

Failure of any gate prevents a stable tag. The already verified candidate may still be published as
`v0.7.0-rc1` with the failed gate stated prominently and no performance-uplift claim.

## 6. Release and repository repair

The v0.4-v0.6 source history currently lives on the PR #4 lineage while `main` points to v0.3.1.
The v0.7 pull request is based on PR #4 source head and targets `main`, bringing the missing source
history back onto the canonical branch.

Release sequence:

1. publish the design and implementation history on `cuj0218/feature-v0.7-evidence-router`;
2. open a pull request to `main` and require CI, type, lint, supply-chain, and secret checks;
3. publish `v0.7.0-rc1` from the reviewed candidate;
4. execute the frozen holdout without changing candidate code;
5. generate JSON, CSV, Korean brief, audit, and responsive visualization;
6. if every promotion gate passes, merge and tag `v0.7.0`; otherwise retain the prerelease only.

Release notes must distinguish observed evidence, uncertainty, and recommendation. “Perfect,”
“proven uplift,” and equivalent claims are prohibited unless an independent external holdout later
supports them.

## 7. Test strategy

- Unit tests cover normalization, action/explanation detection, confidence margin, ties, controls,
  and every contract family.
- Hook contract tests cover raw-first silence, edit observation, first success, no-edit success,
  repeat success, failure, repeated failure, observe mode, forced mode, malformed input, and state
  corruption.
- Property-style mutation tests delete or alter each guard and demonstrate a focused test failure.
- Performance tests execute the local hook repeatedly and assert the p95 budget.
- Package tests assert version, plugin manifest, marketplace metadata, README, release brief, and
  wheel/sdist contents agree.
- The holdout grader tests semantic behavior out of model context and rejects test modifications.
