# Super SOL v0.5 Performance Amplifier design

Date: 2026-07-12  
Status: proposed release-candidate design  
Base revision: Super SOL `0.4.0rc1` at `777c9bd`

## Product thesis

Super SOL v0.5 is a performance amplifier for practical Codex work. Its target is not to unlock a
subscription entitlement, change the intrinsic model, or imitate a Pro account. Its target is to help
a Plus user running a balanced model produce outcomes close to a stronger raw-model reference on
bounded software-engineering tasks.

The headline experiment is:

> Can GPT-5.6 Terra at medium effort with Super SOL v0.5 approach GPT-5.6 Sol at high effort while
> materially outperforming the same Terra/medium configuration without Super SOL?

The product may use instructions and deterministic local tools within one Codex turn. It must not
start another Codex process, call an API directly, switch models, create subagents, or purchase usage.

## Why v0.5 is a different design

The v0.3.1 clean-room result tied raw quality while using 26.86% more tokens and 30.86% more time.
The v0.4.0-rc1 result removed the token penalty but still tied raw quality and used 11.08% more wall
time. The v0.4 candidate therefore failed its preregistered Gate 1 wall-time ceiling and never entered
Gate 2.

Those candidates tested a generic wrapper around the same strong model. They did not test the actual
product thesis of a balanced Plus-usable model plus a task-specific harness against a stronger raw
reference. A generic always-on instruction is withdrawn. v0.5 instead spends attention only on
recognized high-risk failure modes and leaves ordinary work on the stock path.

The SOLTELU holdout is useful only as a routing signal. Its repeated improvement on an authentication
refresh race suggests that concurrency procedures can matter, while its log-normalization and order-
migration regressions show that broad routing is unsafe. v0.5 does not adopt SOLTELU model routing.

## Goals

1. Improve Terra/medium quality on targeted coding tasks relative to Terra/medium raw.
2. Approach the quality of Sol/high raw on the same tasks and fixtures.
3. Keep the ordinary-task path behaviorally equivalent to stock Codex.
4. Keep all activity inside one user-started Codex turn.
5. Require observed tool evidence before a correction instruction can activate.
6. Remain beginner-friendly: installation is the setup; no API key or routing configuration is needed.
7. Produce evidence that separates model, effort, harness, task, grader, time, and token effects.

## Non-goals

- Unlocking ChatGPT Pro features, Pro models, account limits, or priority traffic.
- Claiming that a plugin changes the model's intrinsic intelligence.
- Automatically choosing or switching the user's model.
- Automatically launching another model turn, API request, subagent, or paid benchmark.
- Optimizing for every conversation type or every programming language in v0.5.
- Reusing the disclosed T109-T116 set as an unseen holdout.
- Publishing a general performance claim from a tuning set.

## Benchmark cells

The candidate is evaluated with four fixed cells. Model and effort are explicit inputs, never inferred
from an account label.

| Cell | Model | Effort | Super SOL | Role |
|---|---|---|---|---|
| `terra-raw` | GPT-5.6 Terra | medium | off | Plus-efficient raw baseline |
| `terra-v05` | GPT-5.6 Terra | medium | v0.5 candidate | performance-amplifier candidate |
| `sol-high-raw` | GPT-5.6 Sol | high | off | Pro-grade quality reference |
| `sol-max-raw` | GPT-5.6 Sol | max | off | ceiling reference only |

The public claim uses capability-tier names, not subscription entitlement language. In Codex, eligible
Plus users can also select Sol. Therefore `sol-high-raw` is called a Pro-grade reference, not a
Pro-exclusive account cell. A literal Plus-plan versus Pro-plan usage study is a separate experiment.

The primary causal comparison is `terra-v05` minus `terra-raw`; these cells use the same Codex binary,
model, effort, prompt, fixture, environment, timeout, permissions, grader, and execution order policy.
`sol-high-raw` measures proximity. `sol-max-raw` is descriptive and cannot by itself pass or fail the
candidate.

## Runtime architecture

### 1. Conservative local router

`UserPromptSubmit` runs a deterministic, local classifier over the submitted prompt. It performs no
network call and reads no source file. The classifier returns a route, confidence, and matched signal
identifiers without retaining the prompt.

Initial routes:

- `concurrency_state`: races, locks, cancellation, refresh sharing, async coordination, idempotency;
- `security_boundary`: authentication, authorization, path traversal, archive/upload boundaries,
  secrets, untrusted input, symlinks;
- `migration_compatibility`: schema versions, backward compatibility, unknown-field preservation,
  normalization, rollback, repeat migration;
- `failure_atomicity`: partial failure, checkpointing, transactions, retry classification, validation
  before mutation, ordered error reporting;
- `pass_through`: every request that does not cross a conservative activation threshold.

Each specialist route requires either one unambiguous phrase or two independent weaker signals.
Ambiguous, mixed, explanation-only, and non-action prompts use `pass_through`. The router never
selects a model.

The classifier normalizes with Unicode NFKC and `casefold()`, then matches literal phrase tables.
Unambiguous signals have weight 2 and weaker signals have weight 1. A route activates at weight 2.
If two routes reach the threshold, the request is ambiguous and uses `pass_through`; no precedence
breaks a tie. The signal tables are public versioned data, not learned weights or regular expressions
fitted to hidden tasks.

### 2. Frozen specialist procedure packs

Every active route emits one exact compact procedure pack through `additionalContext`. Each pack is
bounded to 90 English words and contains only:

1. the invariants to state before editing;
2. the failure boundary to inspect;
3. the minimum verification category to run;
4. a rule to repair only an observed failure;
5. a rule to preserve the requested API and avoid unrelated refactoring.

The generic path emits no context. Packs are versioned constants with golden tests. They do not name
benchmark task IDs, fixture paths, expected grader assertions, or model tiers.

The exact v0.5 candidate packs are:

- `concurrency_state`: “Before editing, list the shared state, owner, and per-key synchronization
  invariant. Check same-key coalescing, different-key independence, cancellation, failure cleanup,
  and retry after failure. Preserve the public API. Add or run a deterministic concurrency test.
  Repair only an observed failure; avoid unrelated refactoring.”
- `security_boundary`: “Before editing, identify the trust boundary and each input or path
  normalization step. Validate before mutation; reject traversal, aliases, symlinks, partial writes,
  and secret exposure when applicable. Preserve the public API. Run focused boundary tests. Repair
  only an observed failure; avoid unrelated refactoring.”
- `migration_compatibility`: “Before editing, write source and target invariants for every supported
  version. Preserve unknown and nested fields, input immutability, idempotence, normalization order,
  and future-version rejection. Preserve the public API. Run a focused version matrix. Repair only an
  observed failure; avoid unrelated refactoring.”
- `failure_atomicity`: “Before editing, state atomicity, ordering, and idempotency invariants. Separate
  validation, mutation, and commit; distinguish transient, permanent, and cancellation failures; and
  checkpoint only after success. Preserve the public API. Run focused failure-injection tests. Repair
  only an observed failure; avoid unrelated refactoring.”

### 3. Evidence-triggered correction

The plugin does not start a second model call. Within the active turn, a `PostToolUse` hook observes
only recognized verification commands and their structured exit status. A successful verification
emits no context. The first failed recognized verification may emit exactly: “Verification failed.
Use only the observed failure to revisit the active route invariants, make the smallest correction,
and rerun the same focused verification once.” Later failures emit no additional context. Ordinary
shell commands and file edits emit no context.

Recognized verification commands remain structural allowlist entries such as pytest, Ruff, supported
type checkers, language test commands, and `git diff --check`. Shell chains, comments, masked failures,
and textual mentions do not count as evidence.

The hook state records only schema version, selected route, matched signal identifiers, whether one
repair context was emitted, billing authorization, and monotonic event timestamps. It never stores the
prompt, command output, source content, authentication data, or a secret-shaped value.

### 4. Billing and secret boundary

The existing direct-API and live-evaluation guard remains. A live benchmark requires both the exact
standalone user confirmation and the CLI `--confirm-billable` flag. Dry-run remains available without
authorization. A secret-shaped prompt is blocked without echo or persistence.

Malformed hook input fails open for normal work with a short warning. Missing authorization state
fails closed only for recognized billable paths. No hook retries itself.

## User experience

After installation, automatic routing is enabled. A beginner can ask for an implementation or fix in
ordinary language. Most work remains stock Codex. When a high-confidence route activates, the UI may
show a short status such as `동시성 실패 조건 확인 중`; it must not ask the user to configure a model
or API key.

Two exact control lines are supported at the beginning of an actionable prompt:

- `SUPER SOL OFF`: remove that line, then force the remaining request to `pass_through` for the current
  turn while retaining the billing guard;
- `SUPER SOL ROUTE <route>`: remove that line, then explicitly select one published specialist route
  for the remaining request in the current turn.

Controls appearing inside prose or quoted text have no effect. An invalid first-line route fails closed
to `pass_through` with a warning. Explicit routing is recorded as such so automatic-router and
user-forced results cannot be pooled.

## Data flow

1. Codex submits the user prompt to the local hook.
2. Secret and billing checks run before routing.
3. The router selects `pass_through` or one specialist pack.
4. Codex works in the same turn with no nested model process.
5. Recognized verification results are observed locally.
6. At most one failed verification emits a compact repair context.
7. Sanitized route and evidence metadata may be included in benchmark artifacts by digest.

## Error handling

- Unknown or malformed prompt payload: continue without a pack and emit a local warning.
- Route tie or confidence below threshold: `pass_through`.
- State write failure: continue without a pack; recognized billable commands remain denied.
- Unrecognized verification command: ignore it rather than treating it as a pass or failure.
- Failed verification: emit at most one repair context, never an automatic continuation.
- Rate limit, quota, session limit, timeout, provider failure, malformed telemetry, or grader
  infrastructure failure: mark the slot infrastructure-missing and never assign score zero.
- Missing or duplicate benchmark terminal record: reject the slot.

## Test strategy

### Gate 0: no model calls

Gate 0 must make zero Codex model calls and zero grader-container executions. It includes:

- router table tests for positive, negative, ambiguous, mixed, Korean, and English prompts;
- exact golden tests for all four specialist packs and their 90-word ceiling;
- proof that pass-through prompts emit no additional context;
- one-repair-only tests driven by structured failed verification events;
- masked-failure and shell-chain rejection tests;
- billing, secret, malformed-input, privacy, and concurrent-state tests;
- stock Codex plugin install/list/remove/reinstall in isolated homes;
- clean-room dry-run for all four cells;
- package, lint, type, coverage, dependency, secret, archive, and container-security checks.

### Gate 1: tuning and calibration

Gate 1 uses four disclosed tasks covering the initial routes: T102 safe extraction, T103 retry
idempotency, T104 schema migration, and T112 authentication refresh race. Every task runs in all four
cells with two repetitions, for 32 paid slots. Completed slots are immutable and never automatically
retried. A changed router or pack is a new candidate and does not inherit a prior result. Gate 1 cannot
support a public performance claim.

### Gate 2: new sealed holdout

T109-T116 are disclosed and cannot serve as an unseen v0.5 holdout. Before candidate implementation
is frozen, a separate benchmark repository must contain T117-T124 prompts, fixtures, private graders,
and a digest-pinned grader image. Only task-category labels and aggregate digests enter the candidate
repository. Candidate code must not read the private graders.

Gate 2 uses eight tasks, four cells, and two repetitions: 64 planned paid slots. It runs only if Gate 1
passes and the user gives a fresh standalone billable approval. An independent audit must reproduce
the aggregates from sanitized records and find zero artifact omissions or leakage.

## Promotion metrics

The primary paired unit is task and repetition. Bootstrap confidence intervals use fixed seed
`20260712` and 10,000 resamples of paired units. Four-cell execution uses this balanced order, rotated
by task and reversed for repetition two: `A-B-D-C`, `B-C-A-D`, `C-D-B-A`, `D-A-C-B`, where A is
`terra-raw`, B is `terra-v05`, C is `sol-high-raw`, and D is `sol-max-raw`. A v0.5 candidate must
satisfy all of the following on Gate 2:

1. `terra-v05` mean score minus `terra-raw` is at least +5 points.
2. The paired 95% bootstrap lower bound for that amplification delta is at least 0 points.
3. `terra-v05` full-pass rate is greater than `terra-raw` full-pass rate.
4. `terra-v05` mean score is no more than 2 points below `sol-high-raw`.
5. No task shows a `terra-v05` versus `terra-raw` regression of 10 points or more in both repetitions.
6. Across the full holdout, `terra-v05`/`terra-raw` mean total-token ratio is at most 1.15.
7. Across the full holdout, `terra-v05`/`terra-raw` mean wall-time ratio is at most 1.15.
8. On pass-through tasks alone, both token and wall-time ratios are at most 1.05.
9. Every artifact and leakage audit count is zero.
10. An independently implemented parser reproduces pair counts, routes, scores, pass rates, token
    means, wall-time means, and all primary deltas.

`sol-max-raw` is reported as a ceiling reference and has no promotion threshold. Provider cost is
reported only when the provider exposes it and is never imputed from tokens.

## Claim rules

Before Gate 2 and independent audit, the package is `0.5.0rc1` and documentation says only that it is
a performance-amplifier candidate. Passing Gate 1 permits no public performance language.

If every Gate 2 condition passes, the allowed claim is limited to the tested surface:

> On the published sealed software-engineering holdout, Terra/medium with Super SOL v0.5 improved over
> Terra/medium raw and finished within the preregistered quality margin of Sol/high raw.

The project must not claim universal Pro equivalence, account-limit changes, or intrinsic model
improvement. A failed condition keeps the candidate experimental and blocks a stable v0.5 tag.

## Orca clean-room execution

Orca is the worktree and terminal source of truth for paid slots. Each slot receives a fresh worktree,
an arm-specific minimal `CODEX_HOME`, the same environment allowlist, and exactly one Codex process.
The raw cells contain no Super SOL plugin. The candidate cell contains exactly one immutable v0.5
plugin tree. Authentication is copied with mode 0600 into ephemeral homes and excluded from artifacts.

The runner records the Orca worktree ID, terminal handle, Codex binary/version/digest, model, effort,
plugin revision/tree digest, prompt and fixture digests, exact command template, elapsed time, strict
JSONL telemetry, changed-file digest, grader image digest, and sanitized outcome. UI screenshots are
supporting execution evidence, not the authoritative metric source.

## Release and rollback

- Python package version: `0.5.0rc1`.
- Codex plugin version: `0.5.0-rc1`.
- v0.3.1 remains the stable release until v0.5 passes every gate.
- v0.4.0-rc1 remains an immutable failed candidate and is never retagged.
- Any change to router signals, pack text, hook behavior, task data, model, effort, or grader creates a
  new candidate revision and run identity.
- Disabling the plugin or sending `SUPER SOL OFF` restores the pass-through path immediately.

## Acceptance criteria for implementation

Implementation is ready for a paid Gate 1 only when:

1. every Gate 0 check has been observed passing;
2. the worktree is clean and every candidate file is committed;
3. the four-cell dry-run produces the complete planned lattice and starts no model process;
4. the plugin lifecycle succeeds in a fresh stock Codex home;
5. the sealed-holdout manifest and grader image digests exist outside the candidate repository;
6. a separately implemented audit command that imports no candidate router or report code reports no
   blocking security, leakage, or benchmark-integrity finding;
7. the user provides a new explicit billable-run confirmation.
