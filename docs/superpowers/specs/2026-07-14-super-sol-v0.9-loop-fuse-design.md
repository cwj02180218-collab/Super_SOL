# Super SOL v0.9 Loop Fuse Design

Date: 2026-07-14
Status: approved design, pending written-spec review
Release target: `v0.9.0`, preceded by immutable candidate `v0.9.0-rc1`

## Objective

Ship a Sol-specific circuit breaker that bounds repeated verification, recursive subagent growth,
and no-progress compaction without reducing ordinary coding quality or adding model calls. The
product must preserve the v0.8 raw-first contract and remain passive when no loop evidence exists.

The release claim is deliberately narrow:

> Super SOL v0.9 detects and bounds selected no-progress loops on the tested Codex hook surface.

It must not claim to repair Codex Desktop process management, guarantee that every cancellation
reaches every child process, or increase the underlying model's capability ceiling.

## Failure Taxonomy

The design uses product-agnostic, observable runtime symptoms. Each category is defined by hook
events, repeated actions, and missing progress milestones rather than by any external source
identity, private corpus, transcript, or named orchestration product. Sealed synthetic sequences
reproduce the boundary conditions without importing source material into the product or evidence.

The observed failure family has four members:

- **verification replay:** an already observed result is requested again without an intervening edit;
- **tool replay:** the same operation reaches the same outcome repeatedly without an observable
  milestone;
- **agent recursion:** child agents spawn more agents or the root creates an unbounded child set;
- **compaction replay:** automatic compaction repeats without a milestone between compactions.

## Approaches Considered

### Prompt-only brake

A short Sol instruction could ask the model not to repeat tests or spawn extra agents. This has
negligible engineering cost but cannot enforce a bound when the model or an injected continuation
ignores the instruction. It remains useful only as the one-time soft warning.

### External process watchdog

A desktop process monitor could kill long-running Codex children. It can act when no hook fires, but
it cannot reliably infer ownership from elapsed time alone and may interrupt writes, leave partial
state, or kill an unrelated task. Process termination is excluded from the default product.

### Hook-enforced circuit breaker

This is the selected design. Existing Codex lifecycle and tool hooks provide typed session, turn,
model, tool, subagent, and compaction events. Super SOL keeps a bounded local progress ledger,
warns once, blocks only the repeated action, and terminates supported lifecycle paths after the hard
threshold. No hook starts another model turn.

## Product Boundary

### Model gate

Active loop-fuse behavior applies only when the normalized model identifier is exactly
`gpt-5.6-sol`. Terra, Luna, missing model metadata, malformed metadata, and unknown models remain in
observation-only mode. Existing secret protection and explicit billing authorization are unchanged.

`SUPER SOL OFF` remains a per-turn pass-through control. A new user turn resets the loop ledger.
Internal continuation, mailbox delivery, hook-injected context, compaction, and subagent messages do
not reset it.

### Preserved v0.8 behavior

The v0.8 raw-first evidence router remains intact:

1. prompt submission classifies without model-visible context;
2. a successful edit must be observed;
3. the first recognized verifier result may emit one bounded semantic residual or repair context;
4. later events cannot emit another semantic context.

The loop fuse gets no authority to switch models, raise effort, spawn agents, rerun commands,
continue a stopped turn, or create a new API request.

The v0.8 semantic residual and the v0.9 soft loop warning share one model-visible soft-context
claim. If either has already been emitted, the other is suppressed. A hard action denial may still
return one concise reason because the denied tool result must be actionable; every such reason uses
the same 180-code-point ceiling.

### Explicit non-goal: Codex Desktop cancellation repair

Super SOL runs at the plugin hook layer. It can block tool calls and stop supported compaction
lifecycle paths, but it cannot guarantee repair of the Codex Desktop stop button, app-server queue,
or OS process cleanup. v0.9 ships no automatic process killer. Privacy-safe fuse telemetry records
whether the host confirmed a terminal response, but no Super SOL command terminates a process.

### Runtime capability floor

Active fuse mode requires working `PreToolUse`, `PostToolUse`, `SubagentStart`, `SubagentStop`,
`PreCompact`, and `PostCompact` events plus terminal `continue:false` behavior for compaction hooks.
The initial compatibility target is the desktop-bundled `codex-cli 0.144.0-alpha.4` observed during
design. Gate 0 tests the event contract instead of inferring support from a version string. If the
contract probe fails, installation diagnostics report the missing capability and the product must
not claim active loop protection on that runtime.

## Architecture

The feature is divided into four bounded units.

### Progress ledger

The ledger owns per-session, per-turn state. It stores only:

- schema and product version;
- hashed session and turn identifiers;
- model gate and intervention profile enums;
- current edit epoch;
- bounded action fingerprints and outcome classes;
- verifier success and failure counters;
- active and total subagent counters;
- automatic compaction streak;
- warning and hard-stop claim bits;
- last event monotonic timestamp.

It never stores prompt text, source text, command text, tool output, model output, paths,
environment values, chat messages, or secrets. Action fingerprints use normalized enums plus a
private keyed digest. The ledger retains at most eight fingerprints and one state file remains at
most 4096 bytes. Writes are atomic and locked using the repository's existing state pattern.

### Progress classifier

An event advances the progress epoch only when it observes one of these milestones:

- each successful file-mutation tool call, claimed once by tool-use identifier, advances the edit
  epoch;
- a verifier outcome transition, including a lower known failure count or fail-to-pass transition;
- completion of a requested child agent with a new terminal result class;
- accepted input from the user that starts a new turn.

Reading another file, repeating the same search, waiting on the same children, receiving the same
verifier outcome, or automatic compaction is not progress. The classifier uses structured hook
fields and existing command classification. It does not infer progress from persuasive output text.

### Policy engine

The policy engine consumes the ledger and returns exactly one of four decisions:

- `pass`: no model-visible output and no block;
- `warn_once`: one concise instruction to finalize or choose a new evidence-producing action;
- `block_action`: deny only the repeated tool or child creation request;
- `stop_turn`: return the supported lifecycle stop response with a stable reason code.

Policy decisions are deterministic for the same event sequence. They do not call a model and do not
parse the transcript.

### Hook adapter

The existing dispatcher remains the only plugin entry point. It routes these events:

- `UserPromptSubmit`: initialize a fresh turn ledger after model gating;
- `PreToolUse`: block a repeated verifier, repeated no-progress tool, nested spawn, or exhausted
  child budget;
- `PostToolUse`: classify mutation and verifier outcomes and emit at most one soft warning;
- `SubagentStart` and `SubagentStop`: maintain bounded child lifecycle state;
- `PreCompact` and `PostCompact`: count only automatic no-progress compactions and terminate at the
  hard threshold;
- `Stop`: no handler is installed, so Super SOL never prevents a model from stopping.

## Default Fuse Policy

### Verification replay

- After a verifier passes, the same verifier fingerprint is blocked until a new successful edit.
- After a verifier fails, an identical rerun with no successful edit is blocked.
- An edit advances the epoch and permits a new verification.
- A broader verifier is not treated as identical to a focused verifier, but each has its own
  fingerprint and remains subject to generic replay limits.

The blocking response states that the result is already observed and asks the model to summarize or
make an evidence-producing change. It never asks the model to run another test.

### Generic tool replay

- The first two identical no-progress outcomes pass.
- The third emits the one-time warning.
- The fourth matching request is blocked.
- A progress milestone resets the matching streak for the new edit epoch.

Read-only exploration is fingerprinted by normalized operation class rather than raw path or query,
so no sensitive text enters state. Generic blocking is limited to exact normalized fingerprints;
different operations continue normally.

### Subagent budget

- maximum depth is one: a child cannot create another child;
- maximum concurrent children is two;
- maximum children created in one root turn is three;
- a failed or stopped child still consumes one creation slot;
- `wait`, `send_input`, and resume operations remain available for existing children unless their own
  normalized no-progress fingerprint reaches the generic replay threshold.

The root may continue work after a denied spawn. No automatic replacement child is created. A new
user turn receives a fresh budget.

### Compaction circuit breaker

Only compactions whose trigger is `auto` contribute to the streak. Manual compaction neither counts
nor clears the automatic streak.

- an automatic `PostCompact` with no intervening progress increments the streak;
- any progress milestone clears the streak;
- the third consecutive no-progress automatic `PostCompact` returns `continue:false` with reason
  `loop_fuse_no_progress_compaction`;
- a later `PreCompact` observes the terminal claim and also returns `continue:false`, preventing a
  synthetic continuation from obtaining a fresh counter;
- terminal fuse state survives internal queued work for the same user turn and is cleared only by a
  new accepted user turn.

This deliberately avoids a lifetime cap on healthy long turns. It limits only consecutive
no-progress automatic compactions.

### Elapsed-time signal

Elapsed time is a secondary signal, never the sole basis for a hard stop. At a hook event after 30
minutes with no progress, the one-time warning may fire. At a supported hook event after 45 minutes
with no progress, the event is handled according to its replay or compaction policy. If no hook
event occurs, v0.9 cannot interrupt the running sampler; the release documentation states this
limitation explicitly.

## Termination Semantics

Hard-stop behavior follows these rules:

1. deny the repeated high-risk action when `PreToolUse` can intercept it;
2. provide one bounded finalization instruction at most once;
3. return `continue:false` when `PreCompact` or `PostCompact` supports terminal execution;
4. never enqueue response items, mailbox work, retries, follow-up turns, or replacement agents;
5. preserve files and observed verifier results as they are;
6. allow the model to return a concise partial-completion summary when the host continues sampling;
7. treat a user cancel or normal model stop as terminal and never override it.

The finalization instruction identifies the reason class, asks for completed evidence and unresolved
work, and forbids another tool or agent call. It contains no task text and is at most 180 Unicode code
points.

## Failure Behavior

- Missing or unknown model metadata fails closed for intervention and open for user work.
- Malformed hook input produces no model-visible context and no block.
- Corrupt or oversized state is quarantined, replaced with a fresh bounded ledger, and never copied
  into model context.
- Hook timeout or interpreter failure does not block ordinary work.
- A stale lock is recovered using the existing bounded state mechanism.
- An unknown tool is observation-only until it has an explicit normalized class.
- Missing subagent stop events cannot increase the creation cap; conservative active counts may deny
  a later spawn but cannot create a child.
- A hard-stop response that the host does not honor is recorded as `host_stop_unconfirmed`; the
  plugin does not retry the stop.

## Configuration

The production defaults above are fixed for `v0.9.0-rc1`. Test fixtures may override thresholds
through explicit constructor arguments. Production environment variables do not raise budgets or
disable individual safeguards; only the existing `SUPER SOL OFF` user control disables intervention
for the turn.

This prevents benchmark arms or ambient shell configuration from silently weakening the release
contract.

## Verification Strategy

### Unit and property tests

- table-driven event sequences for every policy transition;
- edit epochs reset verifier and tool replay streaks correctly;
- internal continuation and compaction do not reset terminal state;
- new accepted user turns do reset state;
- the third no-progress auto-compaction stops while healthy compactions continue;
- verifier success and failure replay rules distinguish intervening edits;
- child depth, concurrent count, and total count are enforced independently;
- malformed payload, unknown tool, corrupt state, and stale lock behavior;
- every state file remains at most 4096 bytes under long randomized sequences;
- every model-visible string remains at most 180 Unicode code points;
- Terra, Luna, missing model, and unknown model produce no fuse intervention.

### Hook contract tests

Golden JSON tests cover `PreToolUse`, `PostToolUse`, `SubagentStart`, `SubagentStop`, `PreCompact`,
and `PostCompact` inputs and outputs. A local compatibility smoke test runs against the Codex binary
bundled with the desktop application. It must prove which `continue:false` paths terminate a turn;
documentation must downgrade unsupported paths to action blocking rather than simulated termination.

### Gate 0: free release checks

The candidate must pass:

- the full existing suite and all v0.9 tests;
- at least 90% line coverage;
- Ruff and basedpyright with zero errors;
- package build, dependency audit, tracked-secret scan, and isolated install/remove/reinstall;
- 300 hook invocations with absolute p95 below 100 ms and incremental p95 below 70 ms;
- zero additional API calls, model switches, retries, continuations, or automatic agent creation;
- unchanged v0.8 raw-first outputs for non-loop fixtures;
- exactly one plugin and one skill in a clean `CODEX_HOME`;
- a passing capability probe for all six required events and terminal compaction behavior;
- no `Stop` hook in the shipped manifest.

### Gate 1: adversarial replay

An unbilled deterministic pack replays at least twelve sequences derived from the observed failure
classes:

- passed-test replay;
- unchanged failing-test replay;
- healthy edit-test-edit-test flow;
- repeated search and read flow;
- nested spawn attempt;
- concurrent and total child exhaustion;
- repeated wait or resume;
- three no-progress automatic compactions;
- healthy compaction separated by progress;
- internal continuation after terminal claim;
- corrupt state recovery;
- non-Sol pass-through.

Every loop fixture must reach a bounded terminal or blocked state. Every healthy fixture must retain
its expected actions.

### Gate 2: isolated live confirmation

Stable promotion requires a newly sealed eight-task crossover:

```text
8 tasks x 2 arms (Sol raw, Sol plus v0.9) x 2 repetitions = 32 valid slots
```

Both arms use fresh symmetric `CODEX_HOME` trees, the same Codex version, `gpt-5.6-sol` at the same
effort, identical prompts, permissions, timeouts, task revisions, and no other plugin or skill. A
runtime inventory and configuration digest must fail closed on contamination.

The private grader remains outside the agent context. Infrastructure failures are censored and
linked to one fresh-worktree retry; they are never scored as zero. No product behavior changes after
the task pack is sealed.

Stable promotion requires every gate:

- mean paired score delta at least 0;
- clustered bootstrap 95% lower bound at least -2 points;
- full-pass count at least raw;
- verifier-observation rate at least raw;
- token ratio at most 1.05;
- wall-time ratio at most 1.10;
- no repeated task regression of 10 points or more;
- zero uncensored turns exceeding the slot timeout;
- zero child-budget violations;
- zero repeated verifier calls after an observed pass;
- zero contamination, hidden-test leak, artifact omission, or provenance mismatch.

Gate 2 is billable and must not run without a separate explicit approval that names the live batch.

## Release and Claims

Passing Gate 0 and Gate 1 permits `v0.9.0-rc1` as a GitHub prerelease. The release notes include the
hook compatibility floor, exact defaults, limitations, and a migration path from v0.8.

Passing Gate 2 permits `v0.9.0` stable, a pull request to `main`, an immutable tag, a non-prerelease
GitHub release, and installation instructions pinned to the stable tag. Stable notes may claim
bounded loop prevention and noninferior quality only for the tested Sol/Codex configuration.

Failure of any stable gate leaves the RC published with the negative result. The same holdout is not
reused after product changes. No release may claim that Super SOL fixes all Codex Desktop infinite
loops, repairs process cancellation, or makes Sol intrinsically more capable.

## Explicit Non-Goals

- patching or forking the OpenAI Codex application;
- killing Codex, shell, test, or subagent processes automatically;
- inspecting or persisting prompt or transcript content;
- parsing full transcripts for semantic similarity;
- changing model, effort, or context-window settings;
- adding another reviewer model or hidden API request;
- allowing hooks to continue a stopped turn;
- tuning thresholds after Gate 2 results are visible;
- claiming universal effectiveness beyond the tested Sol hook surface.
