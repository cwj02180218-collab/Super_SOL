# Super SOL v0.9.1 Selective Gate Design

Date: 2026-07-17
Baseline: `v0.9.0-rc1` (`0dbf09cae2903fd65e306534ae273a15a315d807`)
Candidate target: `v0.9.1-rc1`

## Status and Claim Boundary

This document specifies a patch candidate. It does not claim a quality uplift and it does not
promote a stable release. `v0.8.0` remains stable until a frozen `v0.9.1-rc1` candidate passes a
new, sealed confirmatory holdout.

The candidate has two product goals:

1. make the existing semantic routes reach the model only when classification is confident; and
2. preserve raw-like behavior for every other prompt while adding one bounded verification-debt
   reminder only after observable edit behavior warrants it.

It retains the v0.9 Sol loop fuse. It does not add model calls, model switching, subagents,
automatic continuation, automatic retries, or automatic test reruns.

## Evidence That Motivates the Patch

Two independent 240-slot v0.9 runs passed the contamination and provenance audit. Their pooled
paired effects were near zero:

| Profile | Pooled score delta | Two-way 95% CI | Token ratio | Time ratio |
| --- | ---: | ---: | ---: | ---: |
| Sol/high | -0.10 | [-1.74, +1.32] | 1.048 | 1.017 |
| Terra/xhigh | -0.21 | [-1.25, +1.04] | 0.992 | 0.989 |

The direction did not replicate. T213 and T230 each produced a repeated ten-point regression in
one run, but the active v0.9 prompt route was not naturally reachable and no loop event fired.
Those observations therefore cannot justify task-specific prompt tuning.

Code inspection found three product-level causes for the near-zero treatment effect:

- adaptive routing persisted a detected natural route but applied it only when the route was
  explicitly forced;
- prompts without a forced route received no prompt-time context; and
- the prompt hook created an injection claim but ignored whether it won the claim.

An uncommitted local v0.10 carve demonstrated one possible repair, including a generic pack for
every action-shaped prompt. v0.9.1 deliberately does not adopt that generic always-on pack because
the replicated Sol token ratio was already close to the 1.05 limit and Codex raw already performs
substantial verification by default.

## Considered Approaches

### Observation only

Keep only the loop fuse and private telemetry. This has the smallest behavioral risk but cannot
test whether the semantic contracts improve hard tasks because the treatment remains invisible.

### Selective one-shot intervention (selected)

Inject one short specialist contract only for a high-confidence route. Otherwise remain silent at
prompt time. If two successful edits occur after the most recent verifier, emit one short debt
reminder. This activates the intended mechanism while bounding interference and cost.

### Always-on verification contract

Inject a generic verification instruction for every action prompt. This is closest to Fablize's
always-on discipline, but it duplicates stock Codex behavior and creates the largest token and
latency risk. It is rejected for the default mode.

## Model Profiles

The exact normalized identifiers `gpt-5.6-sol` and `gpt-5.6-terra` are eligible for selective
semantic intervention. The state records only the enum `sol`, `terra`, or `observe`, never the raw
model string.

Unknown, missing, malformed, preview, Luna, and future identifiers remain observation-only. A
model-visible response is allowed only when the current event profile exactly matches the profile
sealed at prompt time. Model drift fails closed.

The loop fuse remains active only for exact Sol. Terra receives semantic routing and evidence
discipline but no active loop stop/warn behavior in this patch. This keeps the loop mitigation tied
to the reproduced Sol failure report instead of generalizing it without evidence.

## Prompt-Time Routing

`route_prompt` continues to require an action-shaped request. Explanation-only and non-action
prompts pass through. A specialist route is eligible only when all of the following hold:

- the best contract score is at least 2;
- the best contract beats the runner-up by at least 1;
- the selected route maps to one of the four frozen specialist families; and
- the model profile is exact Sol or exact Terra.

The four route families remain concurrency state, security boundary, migration compatibility, and
failure atomicity. Their text is rewritten to at most 180 Unicode code points each. A pack asks for
only the relevant invariant and one focused verifier. It must not request a printed checklist,
broaden scope, rerun a passing test, or mention a benchmark task.

Ambiguous action prompts and low-confidence action prompts persist a small private action marker so
that later edit evidence can be evaluated, but they emit no prompt context. This is the key
difference from the rejected generic baseline pack.

The dispatcher fast path delegates action-shaped prompts to the full processor so the private
marker can be created. Explanation, ordinary conversation, and no-signal prompts retain the silent
low-import path.

## Evidence-Time Gate

The evidence hook records only successful or failed edit/verifier enums with hashed tool-use
identifiers. It never records source, command text, tool output, prompt text, paths, or secrets.

A verification debt exists when at least two distinct successful edit events have occurred since
the most recent verifier event. On the second such edit, the hook may emit one reminder:

> Two successful edits occurred without a verifier. Run the narrowest relevant check now and read
> its result before editing again.

The reminder is bounded to 180 code points. A duplicate hook delivery for the same tool-use ID does
not count as another edit. A verifier resets the edit-since-verifier count for future debt
decisions, whether the verifier passes or fails.

For a confidently routed prompt, the existing post-verification behavior remains available:

- a failed verifier may emit one repair context based only on the observed failure; or
- a successful verifier may emit one residual semantic boundary check.

Debt, repair, residual, and loop warnings share one evidence channel. The first eligible mechanism
wins; none may trigger automatic continuation.

## Injection Budget and Fail-Closed Behavior

There are two typed model-visible channels per turn:

- `prompt`: at most one specialist route pack; and
- `evidence`: at most one debt, repair, residual, or loop warning.

The absolute maximum is two bounded contexts per turn, one from each channel. Claims are atomic and
their return values are honored. Repeated prompt delivery, repeated tool delivery, hook races, and
compaction cannot create an additional context.

If `PLUGIN_DATA` is unavailable, state is unreadable, the profile drifts, a claim cannot be
created, or a route is malformed, the product continues without model-visible context. Secret
blocking and explicit billable-command denial retain their existing fail-closed behavior.

The private state namespace is versioned so stale v0.9 turn artifacts cannot affect v0.9.1.

## Telemetry and Privacy

Private records contain only:

- schema version;
- profile enum;
- diagnostic mode enum;
- actionable boolean;
- route and contract enums;
- bounded signal identifiers;
- billable authorization boolean;
- hashed session, turn, and tool-use identifiers;
- edit/verifier success booleans; and
- claimed context channel enums.

The implementation must prove through tests that raw prompts, source text, command text, command
output, paths, API keys, and environment secrets are absent. Files are owner-only and bounded.

## Version and Release Contract

The implementation candidate uses:

- Python package version `0.9.1rc1`;
- Codex plugin version `0.9.1-rc1`; and
- Git prerelease tag `v0.9.1-rc1` only after reviewed CI and the free gates pass.

The candidate release brief must say that v0.8.0 remains stable and that v0.9.1-rc1 is an
experimental selective gate. It may report only observed evidence. A stable `v0.9.1` tag is
created only after every confirmatory promotion gate passes.

## TDD and Free Gates

Implementation begins with failing tests for:

1. natural high-confidence routes reaching exact Sol and Terra once;
2. low-confidence, ambiguous, explanation, and unknown-model prompts remaining silent;
3. missing state storage preventing prompt context;
4. prompt claim races allowing one winner;
5. debt appearing after the second distinct successful edit since a verifier;
6. duplicate tool delivery not accruing debt;
7. a verifier resetting the debt sequence;
8. debt, repair, residual, and loop warnings sharing one evidence channel;
9. model drift and diagnostic observe mode remaining silent;
10. every model-visible string staying within 180 code points;
11. no automatic model call, subagent, continuation, retry, or passing-test rerun; and
12. version, package, archive, install, remove, and reinstall consistency.

Gate 0 requires:

- the complete pytest suite with at least 90% coverage;
- Ruff format and lint;
- BasedPyright;
- package archive and isolated plugin lifecycle tests;
- hook latency and deterministic replay;
- privacy, secret, and supply-chain audits; and
- no billable model call.

The v0.9 loop replay must remain behaviorally identical except for intentional schema/provenance
hash changes. Any unreviewed loop-policy change fails Gate 0.

## Confirmatory Holdout

After the candidate commit is frozen, author and seal a new task pack that neither v0.9 nor v0.9.1
has seen. T201-T230 and all earlier tasks are diagnostic-only and cannot support an uplift claim.

The confirmatory lattice is:

- 30 newly sealed tasks, balanced across Python and TypeScript;
- five task families with six tasks each;
- raw and v0.9.1-rc1;
- Sol/high and Terra/xhigh;
- two repetitions; and
- 240 valid slots total.

The pack targets a raw mean between 65 and 85 during independent calibration so there is measurable
headroom. Calibration tasks are discarded before confirmation. Hidden semantic graders remain
outside model context. Slots use fresh sanitized worktrees and `CODEX_HOME` trees, concurrency at
most two, immutable valid records, and linked retries only for infrastructure censoring.

Primary quality is hidden semantic correctness only. Verification behavior, completion, timeout,
provider-native tokens, elapsed time, intervention counts, child sessions, and loop events are
reported separately.

## Promotion Gates

Every stable gate must pass separately in Sol and Terra:

- mean semantic score difference at least 0;
- task-clustered 95% CI lower bound at least -2 points;
- complete-pass rate no lower than raw;
- completion rate no lower than raw;
- paired token ratio at most 1.03;
- paired time ratio at most 1.05;
- timeout and loop-event rates no higher than raw;
- no task with a 10-point or larger regression in both repetitions; and
- zero contamination, hidden-test leak, test-source edit, artifact omission, provenance mismatch,
  or unlinked retry.

Quality-uplift wording additionally requires a mean improvement of at least 2 points and a strictly
positive 95% CI lower bound in the named profile. Passing noninferiority gates does not imply
uplift. Provider-native token values are never compared across profiles.

If either profile misses a stable gate, the candidate remains a prerelease and v0.8.0 remains the
default. The failed holdout is not reused after a code change.

## Expected User Experience

Most turns behave exactly like stock Codex. A user sees no Super SOL narration, checklist, or extra
workflow. On a confidently classified hard task, the model receives one short domain contract. If
the model makes repeated edits without checking its work, it receives one short verifier reminder.
The loop fuse can still warn or stop a genuinely repetitive Sol turn, but the product never starts
another turn on the user's behalf.
