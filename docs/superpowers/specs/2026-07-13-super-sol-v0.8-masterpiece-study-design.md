# Super SOL v0.8 Stable Release and Fablize Comparison Design

Date: 2026-07-13  
Status: proposed; pending written-spec review  
Release target: `v0.8.0`, preceded by immutable candidate `v0.8.0-rc1`

## Objective

Ship the first evidence-bounded stable Super SOL release for GPT-5.6 Sol and produce a
reproducible four-arm crossover study comparing the marginal effects of Super SOL and Fablize.
The release may claim reliability and low overhead only when its preregistered gates pass. It may
claim quality uplift only when the confirmatory quality interval excludes zero in the positive
direction.

The study will also produce an interactive dashboard, publication figures, a Korean executive
summary, and an English arXiv-style paper provisionally titled:

> Do Procedural Harnesses Improve Coding Agents? A Cross-Provider Four-Arm Crossover Study

## Scope Decomposition

The work is executed in three ordered projects. Each project has an immutable handoff to the next.

1. **Product freeze:** refine Super SOL v0.7 into a Sol-gated v0.8 candidate and complete free
   release checks.
2. **Confirmatory experiment:** author a new sealed task pack only after the product commit is
   frozen, then run and audit 96 valid slots.
3. **Publication:** derive every table, figure, and statement from the audited machine-readable
   result bundle.

No product behavior may change after the candidate freeze. A failed confirmatory gate leaves the
candidate as a prerelease and starts a future version; it never triggers post-hoc repair followed by
reuse of the same holdout.

## Product Contract

### Sol-gated active mode

Super SOL reads the Codex hook payload's `model` field. Active semantic intervention is allowed
only when the normalized model identifier is exactly `gpt-5.6-sol`. The confirmatory evidence is
specific to Sol at reasoning effort `high`; documentation must not generalize it to other efforts.

For Terra, Luna, an absent model field, malformed model metadata, or an unknown model identifier,
Super SOL operates in observation-only mode:

- no model-visible context;
- no model switch, subprocess, subagent, retry, or additional API call;
- privacy-safe route telemetry may be recorded using bounded enum values;
- secret-shaped prompt blocking and explicit billable-command authorization remain available.

An explicit per-turn `SUPER SOL OFF` control remains pass-through. Diagnostic forced routing is
test-only and must not bypass the production model gate.

### Raw-first intervention

The v0.7 raw-first state machine remains the behavioral core:

1. Prompt submission classifies at most one semantic contract but emits no context.
2. A successful edit must be observed.
3. The first recognized verification result after that edit may emit one bounded residual or repair
   context.
4. No later event in the turn may emit another context.

The model-visible context ceiling is reduced from 220 to 180 Unicode code points. Context must name
one request-relevant boundary and one observable check. It must not prescribe an implementation,
repeat the task, ask for a checklist, or trigger a second test run.

### Privacy and authority

The plugin persists no prompt text, source text, command text, tool output, model output, environment
value, path, or secret. Stored records contain only schema version, profile enum, contract enum,
signal identifiers, event kind, success boolean, and one-shot claim state. Each state file remains
at most 4096 bytes.

Routing authority does not imply deployment, messaging, purchase, destructive-command, or billing
authority. Known Super SOL live-evaluation commands require the exact standalone approval line
already defined by the plugin.

## Failure Behavior

- Missing or unknown model metadata fails closed for intervention and open for the user's work.
- Malformed hook input returns a concise warning and never injects procedural context.
- State corruption starts a fresh bounded turn state and records no raw payload.
- Hook timeout or runtime failure must not block ordinary editing or verification.
- Secret-shaped prompt input and unapproved known billable commands retain the existing explicit
  denial behavior.
- Infrastructure failures during benchmarking are censored observations, never zero scores.
- Censored slots are rerun once in a fresh worktree with an incremented attempt linked to the
  original record. Further recurrence pauses the batch for diagnosis.

## Candidate Gate 0

Before task-pack authoring, the immutable `v0.8.0-rc1` candidate must satisfy all of the following:

- full Python suite passes;
- plugin-focused suite passes;
- line coverage is at least 90%;
- Ruff and basedpyright report zero errors;
- wheel and sdist build reproducibly and contain no forbidden runtime artifacts;
- production dependency audit reports no known vulnerability;
- tracked secret scan reports zero production matches;
- isolated install, remove, and reinstall leave exactly one plugin and one skill;
- 300 real hook invocations keep absolute p95 below 100 ms and incremental p95 below 70 ms;
- Sol emits at most one context only after edit plus verification;
- Terra, Luna, missing model, and unknown model emit zero contexts across the same lifecycle;
- all emitted contexts are at most 180 Unicode code points;
- GitHub CI and container-security workflows pass for the frozen commit.

## Four-Arm Crossover

### Arms

| Arm | Provider surface | Model and effort | Added product |
| --- | --- | --- | --- |
| `codex-raw` | Codex CLI | `gpt-5.6-sol`, `high` | none |
| `codex-super-sol` | Codex CLI | `gpt-5.6-sol`, `high` | `v0.8.0-rc1` |
| `claude-raw` | Claude Code | `claude-fable-5`, `high` | none |
| `claude-fablize` | Claude Code | `claude-fable-5`, `high` | Fablize `v2.1.0` |

Model and effort must be identical within each provider pair. Provider surfaces remain separate:
raw token totals and dollar costs are not treated as directly interchangeable across providers.

### Clean rooms

Each slot receives a new Orca worktree and a fresh agent session. Raw homes contain authentication
plus the minimum shared configuration and no plugin, skill, MCP server, browser integration, global
instruction, or session history. Candidate homes are byte-for-byte derived from their raw partner,
then receive exactly one pinned product. Runtime inventory and tree digests are recorded before
execution and fail closed on contamination.

Both arms receive the same task prompt, repository revision, public tests, permissions, timeout,
locale, environment allowlist, and shared two-line autonomy and verification instruction. Fablize is
invoked through its documented product surface, not by copying prompt text. Super SOL is invoked by
the installed Codex plugin without benchmark-only forced routing.

### New sealed task pack

T125-T136 are authored after the candidate commit and before any paid slot. The twelve tasks cover:

1. reproducible diagnosis from conflicting symptoms;
2. multi-story completion with cross-file acceptance criteria;
3. render or executable artifact verification;
4. concurrent state and cancellation;
5. backward-compatible migration;
6. path and secret boundary handling;
7. CLI error and exit semantics;
8. pagination under mutation;
9. cache ownership and expiration;
10. atomic publication after partial failure;
11. graph scheduling and cancellation fairness;
12. packaging and release provenance.

Each task has public fixtures and a private semantic grader. Private graders remain outside model
worktrees and are copied into temporary grading checkouts only after the agent exits. The pack,
grader, target, prompt, product pins, and schedule are sealed by commit and SHA-256 digest.

### Schedule

Twelve tasks multiplied by four arms and two repetitions produce 96 valid slots. A balanced Latin
schedule rotates arm and task order across 24 waves. No valid completed slot is retried. Provider
quota or session-limit failures pause only the affected provider pair while preserving paired order.

## Scoring and Statistical Analysis

Each task receives a deterministic 0-100 score:

- public behavior: 20;
- private semantic contract: 50;
- compatibility and regression: 15;
- observed verification after mutation: 10;
- focused change surface: 5.

The primary estimands are within-provider paired effects:

```text
Super SOL effect = score(codex-super-sol) - score(codex-raw)
Fablize effect   = score(claude-fablize) - score(claude-raw)
```

For each product report mean and median paired delta, 95% paired bootstrap interval, exact sign
test, paired rank-biserial effect, wins/losses/ties, full-pass difference, token ratio, wall-time
ratio, changed-line ratio, and verifier-observation rate. Bootstrap resampling is clustered by task
so the two repetitions do not masquerade as independent tasks.

Cross-product comparison uses the difference between the two within-provider effects and the
difference between log token ratios. Raw provider totals are shown but not used as a price or token
winner. Provider-reported USD is reported only where available and never imputed.

### Super SOL stable-promotion gates

Every gate must pass:

- mean paired score delta is at least 0;
- clustered bootstrap 95% lower bound is at least -2 points;
- full-pass count is at least raw;
- token ratio is at most 1.03;
- wall-time ratio is at most 1.10;
- no task repeats a regression of 10 points or more;
- verifier-observation rate is at least raw;
- 96/96 valid slots exist after linked infrastructure retries;
- hidden-test leak, artifact omission, contamination, provenance mismatch, and secret count are zero.

Passing these gates permits the stable claim "noninferior quality with bounded overhead." A quality
uplift claim additionally requires a positive mean, a clustered bootstrap lower bound above zero,
and no full-pass regression. Otherwise documentation explicitly states that uplift was not proven.

## Visualization

The final interactive HTML dashboard and static publication figures share one generated data model.
They include:

1. score versus normalized token ratio with a Pareto frontier;
2. a forest plot of within-provider effects and 95% intervals;
3. a task-by-arm score heatmap with repetition markers;
4. token composition and wall-time ratio panels;
5. full-pass and verifier-observation rate comparison;
6. an infrastructure-censoring and provenance audit panel;
7. a promotion-gate table with machine-derived pass or fail labels.

Desktop and mobile screenshots must be checked for clipping, overlap, unreadable labels, and false
visual precision. Static paper figures are exported as vector PDF or SVG plus 2x PNG fallbacks and
must remain legible in grayscale.

## Publication Artifacts

The publication bundle contains:

- English arXiv-style LaTeX source and compiled PDF;
- Korean executive summary;
- anonymous-ready paper variant without local paths or credentials;
- machine-readable preregistration, slot CSV, analysis JSON, audit JSON, and figure data;
- a reproduction README with exact pinned commands and expected digests;
- a limitations section covering provider token-accounting differences, model-family confounding,
  task-pack scope, repeated measures, researcher-built products, and lack of capability-ceiling
  inference.

The paper distinguishes three claims: procedural adherence, end-to-end agent performance, and
underlying model capability. The experiment can support only the first two. It cannot show that a
harness changes the model's capability ceiling.

## Release Decision

If Gate 0 and every confirmatory promotion gate pass, merge the candidate through a reviewed pull
request, tag `v0.8.0`, publish a non-prerelease GitHub release, and update the stable installation
instructions from `v0.3.1` to `v0.8.0`.

If any promotion gate fails, keep `v0.8.0-rc1` as a prerelease, publish the complete negative result,
leave `v0.3.1` as stable, and do not use "masterpiece," "performance amplifier," or equivalent uplift
language in public documentation.

## Explicit Non-Goals

- combining Super SOL and Fablize in one arm;
- adding SOLTELU, LazyCodex, browser plugins, MCP packs, or global skills;
- changing model or effort during a slot;
- automatic retries, subagents, or hidden extra model calls;
- tuning against T125-T136 after results are visible;
- claiming a universal winner from provider-specific token accounting;
- claiming that either harness raises underlying model intelligence.
