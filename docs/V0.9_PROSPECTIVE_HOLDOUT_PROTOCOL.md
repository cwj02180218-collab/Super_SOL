# Super SOL v0.9 prospective holdout protocol

Status: preregistration draft; no billable slot is authorized by this document

## Purpose

Decide whether the frozen `v0.9.0-rc1` bytes can become stable without tuning on T149-T160, or
whether new data identify a repeated defect that justifies `0.9.1`.

This study measures semantic correctness separately from verification behavior. It is not designed
to manufacture a positive result by selecting tasks after observing Super SOL output.

## Frozen treatment

- Raw arm: stock Codex with no product plugin or unrelated user configuration
- Product arm: `v0.9.0-rc1` at commit `0dbf09cae2903fd65e306534ae273a15a315d807`
- Profiles: GPT-5.6 Sol/high and GPT-5.6 Terra/xhigh
- Codex version, permissions, timeout, prompt, target commit, and base tools: identical within profile
- Product source, prompt router, loop fuse, thresholds, and package bytes: immutable after sealing

## Task construction

Author 40 bounded repository-repair tasks after the treatment freeze. Use 10 tasks only for
difficulty calibration and discard them permanently from confirmation. Seal the remaining 30
before any confirmatory arm runs.

The 30-task holdout contains 15 Python and 15 TypeScript repositories, with six tasks in each
family:

1. concurrency and shared state
2. failure atomicity and transaction boundaries
3. migration and backward compatibility
4. security and path/input boundaries
5. multi-file integration, CLI, and deterministic algorithms

Difficulty is based on contract structure and independent pilot evidence, not on the difference
between raw and Super SOL. Long runtime by itself is not a difficulty criterion. Each task must
remain gradeable at the external cutoff.

## Lattice and execution

- 30 sealed tasks
- 2 products: raw and Super SOL
- 2 model profiles: Sol/high and Terra/xhigh
- 2 repetitions
- Total: 240 valid slots
- Counterbalanced wave order with every cell occupying every ordinal equally
- Fresh physical `CODEX_HOME` and fresh worktree for every slot
- Concurrency at most 2
- Model cutoff 900 seconds; external semantic grader cutoff 120 seconds
- Valid records immutable; only infrastructure-censored attempts receive linked retries

No unrelated plugin, skill, global instruction, cached product path, or hidden grader may appear in
model context. Child sessions and their provider-reported usage are charged to the originating arm.

## Outcomes

Primary outcome:

- Hidden semantic correctness score, 0-100, containing no verification, focus, style, or process points

Secondary outcomes:

- complete hidden pass
- task-level semantic contract coverage
- observed verification, reported separately from quality
- completion before cutoff and timeout
- provider-native tokens within the same profile
- elapsed time
- child/session count and repeated-state events

Timeout code is graded at cutoff. Timeout is not converted to zero; completion and timeout remain
separate operational outcomes.

## Analysis

Pair raw and Super SOL by task and repetition within profile. Use deterministic bootstrap intervals
clustered by task and report wins, losses, ties, and task-family effects. Do not pool raw scores or
provider-native tokens across profiles. Do not combine these results with T149-T160 or Fablize.

## Stable promotion gates

Every item must pass in both profiles:

- mean semantic score difference at least 0
- task-clustered 95% CI lower bound at least -2 points
- complete-pass rate no lower than raw
- completion rate no lower than raw
- paired token ratio at most 1.05
- paired time ratio at most 1.10
- no task with a 10-point or larger regression in both repetitions
- zero contamination, hidden-test leak, artifact omission, provenance mismatch, or unlinked retry

Quality uplift wording additionally requires mean semantic improvement at least +2 points and a
strictly positive 95% CI lower bound in the named profile. Stable promotion does not imply uplift.

## Version decision

- If every stable gate passes: promote the same frozen behavior to stable without tuning on this holdout.
- If a semantic defect repeats in at least two independent new tasks: record the pattern, create
  `0.9.1`, and evaluate it on newly authored replacement tasks that neither version has seen.
- If neither condition holds: keep `v0.9.0-rc1` as prerelease and `v0.8.0` as stable.

The 30 confirmatory tasks become permanently observed after the first valid slot. They may be used
for regression diagnosis but never again for a performance-uplift claim.

## Authorization boundary

This protocol does not authorize live or billable execution. A separate owner instruction must
name the 240-slot batch and explicitly approve billable execution before the first model slot.
